2 # README
3 This directory (_data_) contains:
4 - module test data
5 - The _dmr++_ tools: scripts, source code, and production rules for tools
6 that can be used to create and process `hdf5`/`netcdf-4` data files to create
7 portable _dmr++_ files whose binary data objects are held in a web object store
8  like AWS S3.
9 
10 ## Overview
11 We have developed an initial set of tools that enable a data provider to
12 easily serve data stored in Amazon's S3 Web Object Store. In the current
13 implementaiton, the data must be stored in HDF5 or NetCDF4 files. The data
14 do not, however, have to be reformatted to be used with the Hyrax server.
15 Furthermore, the data objects are subset 'in-place' from S3 instead of first
16 transferring the object and then serving it, resulting in lower response
17 latency than other solutions for S3 darta access such as those based on FUSE
18 filesystems. For data users, access is seamless - there is no difference between
19 access to data stored in S3 or on spinning disk.
20 
21 We have conducted tests of this software and the Google Cloud Store and
22 found that it works with that Web Object Store as well. In fact, reconfiguration
23 to GCS is trivial.
24 
25 
26 ### dmr++
27 The _dmr++_ files are the control data used by the server to enable
28 'in-place' access and subsetting of data in S3.
29 
30 **self-contained and portable**: The _dmr++_ files are self contained. They can
31 be served by any Hyrax server (version 1.16.0 or higher) simply by placing them
32 in the server's data file system.
33 
34 **size**: The _dmr++_ files are typically very much smaller than their source
35 `hdf5`/`nectdf-4` files, by as much as 2 or even 3 orders of magnitude (YMMV).
36 
37 ## dmr++ tools
38 
39 There are three programs for building _dmr++_ files:
40 - The program `get_dmrpp` builds a single _dmr++_ file from a single
41 `netcdf-4`/`hdf5` file.
42 - The program `ingest_filesystem` builds a collection of _dmr++_ files
43 from from data held in the locally mounted filesystem.
44 - The program `ingest3bucket` builds a collection of _dmr++_ files
45 from from data held in Amazon's S3 storage.
46 
47 NOTE: Organizationally, this directory (_data_) and it's child directory
48 _dmrpp_ are arranged in this hierarchy in order to mimic the deployment
49 structure resulting from running "make install". Most modules do not
50 need to do this but since _dmr++_ files reference other files, and they
51 do so using paths relative the BES Catalog Root the mimicry is required.
52 
53 NOTE: Examples can be run as shown from the _bes/modules/dmrpp___module/data_
54 directory.
55 
56 ## building the software
57 
58 In order for these programs (shell scripts) to function correctly a
59 localization step must take place. This happens when the parent software
60 (the `dmrpp_module`) is built and installed as part of the BES. Once this
61 is done the scripts will have been installed and should be in `$prefix/bin`
62 and ready to use.

64 ## `get_dmrpp` - build a _dmr++_ file from an _hdf5_/_nectdf-4_ file.
65 
66 The `get_dmrpp` shell script generates a single _dmr++_ file from a single
67 netcdf-4/hdf5 file. It is used by both `ingest_filesystem` and `ingest_s3bucket`.
68 
69 ```
70  Usage: get_dmrpp [options] <hdf5_file>
71 
72  Write the DMR++ for hdf5_file to stdout
73 
74  By default the BES Data Root directory is set to the CWD. This
75  utility will add an entry to the bes.log specified in the
76  configuration file. The DMR++ is built using the DMR as returned
77  by the HDF5 handler, using options as set in the bes
78  configuration file found here.
79 
80  -h: Show help
81  -v: Verbose: Print the DMR too
82  -V: Very Verbose: print the DMR, the command and the configuration
83  file used to build the DMR
84  -r: Just print the DMR that will be used to build the DMR++
85  -u: The binary object URL for use in the DMR++ file
86  -d: Data root directory for the BES.
87  -o: The name of the file to create.
88 
89  Limitations:
90  * The pathanme to the hdf5 file must be relative from the
91  directory where this command was run; absolute paths will not work.
92  * The build_dmrpp command must be in the CWD.
93  * The bes conf template has to build by hand. jhrg 5/11/18
94 ```

138 ## `ingest_filesystem` - building _dmr++_ files from local files.
139 The shell script `ingest_filesystem` is used to crawl through a branch of
140 the local filesystem, identifying files that match a regular expression
141 (default or supplied), and then attempting to build a _dmr++_ file for each
142 matching file using the `get_dmrpp` program.
143 
144 ```
145  Usage: ingest_filesystem [options]
146 
147  Crawl filesystem and make a DMR++ for every dataset matching the
148  default or supplied regex.
149 
150  The DMR++ is built using the DMR as returned by the HDF5 handler,
151  using options as set in the bes configuration file used by get_dmrpp.
152 
153  -h: Show help
154  -v: Verbose: Print the DMR too
155  -V: Very Verbose: Verbose plus so much more!
156  -j: Just print the DMR that will be used to build the DMR++
157  -u: The base endpoint URL for the DMRPP data objects. The assumption
158  is that they will be organized the same way the source dataset
159  files below the "data_root" (see -d)
160  (default: file://${data_root})
161  -d: The local filesystem root from which the data are to be ingested.
162  The filesystem will be searched beginning at this point for files
163  whose names match the dataset match regex (see -r).
164  (default: $CWD)
165  -t: The target directory for the dmrpp files. Below this point
166  the organization of the data files vis-a-vis their "/" path
167  separator divided names will be replicated and dmr++ files placed
168  accordingly.
169  (default: $CWD)
170  -r: The dataset match regex used to screen the base filesystem
171  for datasets.
172  (default: "^.*\\.(h5|he5|nc4)(\\.bz2|\\.gz|\\.Z)?$")
173  -f: Use "find" to list all regular files below the data root directory
174  and store the list in "${ALL_FILES}" The the dataset match regex is applied
175  to each line in ${ALL_FILES} and the matching data files list is placed in
176  "${DATA_FILES}". If this option is omitted the files named in "${DATA_FILES}"
177  (if any) will be processed.
178  (default: Not Set)
179 ```

If found `ingest_filesystem` will treat every line in `./data_files.txt` as a fully qualifed path to an `hdf5`/`netcdf-4` file for which a `dmr++` file is to be computed. By default the output tree will be placed in the current working directory. The base end point for the `dmr++` binary object will be set to the current working directory.

228 ## `ingest_s3bucket` - building _dmr++_ files from files held in S3.
229 The shell script `ingest_s3bucket` utilizes the AWS CLI to list the contents of an S3 bucket. The name of each object in the bucket is checked against either the defaukt or user supplied regex. Each matching file is retrieved from S3 and then a _dmr++_ is built from the retrived data object. Once the _dmr++_ file is built the downloaded object is deleted unless otherwise instructed. The code relies on the AWS CLI being installed and configured using the `aws configure` command (or it's equivalent).
230 
231 ```
232  Usage: ingest_s3bucket [options]
233 
234  List an AWS S3 bucket and make a DMR++ for every dataset matching the
235  default (or supplied) regex.
236 
237  The DMR++ is built using the DMR as returned by the HDF5 handler,
238  using options as set in the bes configuration file used by get_dmrpp.
239 
240  -h: Show help
241  -v: Verbose: Print the DMR too
242  -V: Very Verbose: Verbose plus so much more. Your eyes will water from
243  the scanning of it all.
244  -j: Just print the DMR that will be used to build the DMR++
245  -s: The endpoint URL for the S3 datastore.
246  (default: https://s3.amazonaws.com)
247  -b: The S3 bucket name.
248  (default: cloudydap)
249  -d: The "local" filesystem root for the downloaded data.
250  (default: ./s3_data/bucket_name})
251  -t: The target directory for the dmrpp files. Below this point
252  the structure of the bucket objects vis-a-vis their "/" path
253  separator divided names will be replicted and dmr++ placed into
254  it accordingly.
255  (default: $CWD)
256  -f: Retrieve object list from S3 bucket into the list file for the bucket,
257  apply the dataset match regex to the object names to create
258  the data files list. If this is omitted the files named in an
259  exisiting bucket list file (if any) will be processed.
260  (default: Not Set)
261  -r: The dataset match regex used to screen the filenames
262  for matching datasets.
263  (default: `^.*\\.(h5|he5|nc4)(\\.bz2|\\.gz|\\.Z)?$`)
264  -k: Keep the downloaded datafiles after the dmr++ file has been
265  created. Be careful! S3 buckets can be quite large!
266 
267 Dependencies:
268 This script requires that:
269  - The bes installation directory is on the PATH.
270  - The AWS Commandline Tools are installed and on the path.
271  - The AWS Commandline Tools have been configured for this user with
272  AWS access_key_id and aws_secret_access_key that have adequate permissions
273  to access the target AWS S3 bucket.
274 
275 ```

284 and the software caches bucket information in the files named in the patterns `s3_BUCKETNAME_all_files.txt` `s3_BUCKETNAME_data_files.txt`. Changing the bucket name will change the name of the file information files accordingly). If the file is found, `ingest_s3bucket` will treat the 4th column of every line in `./s3_cloudydap_data_files.txt` as a relative path to an `hdf5`/`netcdf-4` file in the default bucket (`cloudydap`) for which a `dmr++` file is to be computed. By default the output tree will be placed in the current working directory. The base end point for the `dmr++` binary object will be set the URL of the S3 binary file that
285 was used to create the `https://s3.amazonaws.comdmr++` file.

313 ## What To Do
314 
315 1. Determine the base URL of the hosting service (AWS S3, Google Cloud Store, Apache httpd, etc) you
316 intend to utilize. For example in S3 you need to know the _bucket name_ and the AWS S3 URL (https://s3.amazonaws.com)
317 1. If you haven't already done so put your data into the hosting service space (And if you _do_ have it locally
318 don't remove it until the _dmr++_ process is completed!)
319 1. Use the tools discussed below to make one or dmr++ files for your data (you'll need the bucket name and all that from previosuly)
320 1. Place the _dmr++_ files, in any organization desired, into the BES.Catalog.catalog.RootDirectory tree.
321 1. You can now access the remote data by navigating to the location of the _dmr++_ files and access them through the Data REquest Form or the DAP2 and DAP4 protocols.
