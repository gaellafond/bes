20 # Introduction
21 The Back-end Server (BES) for Hyrax is a unix daemon that builds DAP2 and DAP4 response
22 for various kinds of data. Since the daemon runs on Unix hosts, it often works with data
23 that are stored in files or collections of files found on file systems. However, the BES
24 can also use data stored in database systems, web object stores (e.g., S3), other kinds
25 of web APIs and remote data accessed using plain HTTP.
26 
27 In addition to the DAP2/4 protocols, the BES can package responses to queries for data in
28 a number of well-known binary file types, including NetCDF3/4, GeoTIFF and JPEG2000.
29 
30 All of the functionality specific to DAP or particular types of data is implemented using
31 a group of 'plugin modules.' These modules isolate the operations for specific kinds of
32 daa from the BES software itself. Each kind of data that can be read is accessed using
33 a different module and each response other than the DAP2/4 responses is returned its own
34 module.
35 
36 The BES does not contain (much) software that implements the DAP2/4 protocols. Instead it
37 uses the libdap4 library for that. See [github.com/opendap/libdap4](github.com/opendap/libdap4).
38 
39 The BES does not implement the WEB API for DAP2/4, instead the OLFS (OPeNDAP Lightweight Front-end Server)
40 is used to do that. See [github.com/opendap/olfs](github.com/opendap/olfs).
41 
42 The BES framework is designed to be extensible and can be used to combine reading various
43 kinds of data with building different kinds of responses. The framework is designed to
44 support lazy data read operations, only reading those data that are needed and only when
45 they are needed. Note that some kinds of well-known binary responses must be built in full
46 before they are returned, but this is not a requirement of the framework but the responses,
47 or their APIs.
48

51 latest and most comprehensive Hyrax documentation.</a>

61 # What's here
62 Here there's a bes-config script which will be installing in $prefix/bin that
63 can be used to determine where the libraries and their header files have been
64 installed. This directory also contains some documentation files.
65 
66 _dispatch_: This is where the bulk of the BES source code resides.
67 server: The BES server and standalone executables; build using dispatch
68 cmdln: A command line client which can communicate with the bes server.
69 
70 _dap_: A module that implements the (OPeN)DAP access.
71 
72 _modules_: All the modules that are part of a standard Hyrax distribution.
73 The configure script looks for their dependencies and only tries to build
74 the ones that can be built. Note that the HDF4 and 5 modules are pulled in
75 from separate _git_ repositories.
76 
77 _xmlcommand_: The BES/dispatch software is a framework. The basic commands
78 recognized by it are defined and implemented by software in this directory.
79 
80 _standalone_: The 'standalone version of the server; used for testing.
81 
82 _apache_: An Apache module, currently not part of the default build
83 
84 _ppt_: The PPT implementation. The BES uses PPT for its communication
85 
86 _templates_: A collection of source files which can be used as templates when
87 you write your own handlers/modules, et cetera.
88 
89 _conf_: Where the automake and autoconf configuration files live
90 
91 _docs_: Where some bes documentation resides

93 # Configuration
94 The BES configuration is controlled by a set of configuration files. While the complete
95 configuration could be held in one file, it is easier to store information
96 specific to each module in a different file. The main configuration file then
97 reads all of the module configuration files, following a common Unix pattern.
98 
99 ## Basic Configuration
100 Once the BES software has been installed, you will need to make a few
101 changes to the BES configuration file, which is the bes.conf located in
102 _prefix_`/etc/bes/bes.conf`. Module and handler configuration files will
103 be installed in the _prefix_`/etc/bes/modules` directory.
104 
105 Only a few parameters need to be modified to get the BES up and running
106 for the first time. These parameters are located at the top of the
107 configuration file are are:
108 
109 **BES.ServerAdministrator**
110 - set this to an email address that can be used for users to contact if
111  there are issues with your installation of the server.
112 
113 **BES.User=user_name**
114 
115 **BES.Group=group_name**
116 - set these to a valid username and groupname on your system. We
117  recommend that you create a username and groupname called bes that has
118  permissions to write only to the BES installation directory. We'll
119  need to write to the log file, and that's about it.
120 
121 **BES.LogName=./bes.log**
122 - Set this to the full path and file name for where you want the BES log
123  file to be written.
124 
125 With this configuration you will be able to start the BES. No handlers
126 or modules have been installed yet, so you won't be able to serve data,
127 but the BES should run.
128 
129 ## Configuration for Hyrax
130 For the BES to run with Hyrax, additional changes will need to be made
131 to the dap.conf and dap-server.conf files, which are located in the
132 modules directory _prefix_`/etc/bes/modules`. The dap module should be
133 installed by default with the BES. If the dap.conf file is not found, be
134 sure that you have libdap installed. The dap-server module, known as the
135 General purpose handlers, adds responses for the DAP ascii response, the
136 DAP info response, and the DAP html form response. You will need to get
137 the dap-server module. You should see a dap-server.conf file in the
138 modules directory.
139 
140 The changes required for Hyrax are:
141 
142 **BES.Catalog.catalog.RootDirectory** = _path_to_root_data_directory_
143 - Find the BES.Catalog.catalog.RootDirectory parameter in dap.conf. Set
144  this to the root directory of your data. If you're serving data stored
145  in files, this is the place in the file system where those files are
146  stored.
147 
148 **BES.Catalog.catalog.Include**=;
149 - This parameter specifies what files/directories are included in the
150  list of nodes in the catalog. The default is to show everything. After
151  this parameter is looked at, the Exclude parameter is then looked at
152  to see what files you might want to exclude.
153 
154 __BES.Catalog.catalog.Exclude=^\..*;__
155 - This parameter specifies what files/directories to include in the list
156  of nodes in the catalog. The default, shown here, is to exclude any
157  files or directories that starts with a dot (.)
158 
159 The only possible configuration parameter that you may need to change is
160 the one that maps a file to a data handler. This parameter is called
161 BES.catalog.TypeMatch, and is found in each of the data handler
162 configuration files, such as nc.conf. The default values should work.
163 
164 The value of this parameter is a semicolon separated list that matches
165 the name you used in the BES.modules parameter with different datasets.
166 The BES uses regular expressions to identify different types of
167 datasets. In the example given in the file, any dataset name that ends
168 in '.nc' will be accessed using the netcdf hander (because the name 'nc'
169 is used here with the regular expression '.*\.nc'; note that the BES
170 uses regular expressions like Unix grep, not file globbing patterns like
171 a command shell). Since the name 'nc' was associated with the netcdf
172 module in the modules section, the netcdf module will be used to access
173 any dataset whose name that ends in '.nc'.
174 
175 The regular expressions shown in the examples are simple. However, the
176 entire dataset name is used, so it's easy to associate different modules with
177 datasets based on much more than just the 'file name extension' even though
178 that is the most common case.
179 
180 To test your regular expression for the TypeMatch parameter, or the
181 Include and Exclude parameters, use the supplied besregtest program.
182 Simply run besregtest to discover its usage.
183 
184 ### Installing a custom handler/module
185 By default, The BES for Hyrax comes with a suite of handlers that read
186 a number of data formats. However, you can install custom handlers that
187 are not distributed by default. I'll use the SQL handler as an example.
188 
189 Get the SQL handler source code from http://github.com/opendap/,
190 making sure that the version supports Hyrax. Expand the tar.gz file
191 and follow the instructions with the following caveat:
192 If you have installed the BES using a prefix other than /usr/local (the
193 default), make sure that the correct bes-config is being run. If you are
194 having problems compiling or linking the handler, try using not only
195 --prefix=... but also --with-bes=... when you configure the handler.
196 
197 #### Install the newly-built handler.
198 Once built, install the handler using 'make install'. This will install
199 the BES module and a configuration file to use for that module. Each
200 module will have its own configuration file. In this case it is nc.conf
201 and installed in the _prefix_`/etc/bes/modules` directory. The next time
202 the BES is run, this configuration file will be read and the netcdf
203 module loaded. No modifications are necessary.
204 
205 ## Testing
206 To test the server, open a new terminal window and start the bes by using
207 the bes control script besctl, which is installed in _prefix_`/bin`.
208 using the -c switch to name the configuration file. If the server standalone
209 starts correctly it should print something like the following to stdout:
210 
211 ```
212  [jimg@zoe sbin]$ besctl start
213  BES install directory: <prefix>/bin
214  Starting the BES daemon
215  PID: <process_id> UID: <uid of process>
216 ```
217 
218 Go back to your first window and run the bescmdln program. Use the -h (host)
219 and -p (port) switches to tell it how to connect to the BES.
220 
221 ```
222  [jimg@zoe bin]$ bescmdln -p 10002 -h localhost
223 
224  Type 'exit' to exit the command line client and 'help' or '?' to display
225  the help screen
226 
227  BESClient>
228 ```
229 
230 Try some simple commands:
231 
232 ```
233  BESClient> show help;
234  <showHelp>
235  <response>
236 
237  ...
238 
239  </response>
240  </showHelp>
241  BESClient>
242 ```
243 
244 Note that all commands must end with a semicolon except 'exit'.
245 
246 Now try to get a DAP response using a handler you registered (the BES
247 supports a fairly complex syntax which I won't explain fully here):
248 
249 ```
250  BESClient> set container in catalog values n1, data/nc/fnoc1.nc;
251  BESClient> define d1 as n1;
252  BESClient> get das for d1;
253  Attributes {
254  u {
255  String units "meter per second";
256  String long_name "Vector wind eastward component";
257  String missing_value "-32767";
258  String scale_factor "0.005";
259  }
260  v {
261  String units "meter per second";
262  String long_name "Vector wind northward component";
263  String missing_value "-32767";
264  String scale_factor "0.005";
265  }
266  lat {
267  String units "degree North";
268  }
269  lon {
270  String units "degree East";
271  }
272  time {
273  String units "hours from base_time";
274  }
275  NC_GLOBAL {
276  String base_time "88- 10-00:00:00";
277  String title " FNOC UV wind components from 1988- 10 to 1988- 13.";
278  }
279  DODS_EXTRA {
280  String Unlimited_Dimension "time_a";
281  }
282  }
283  BESClient> exit
284  [jimg@zoe bin]$
285 ```
286 
287 If you got something similar (you would have used a different dataset name
288 from "data'nc'fnoc1.nc" and thus would get a different DAS response) the BES
289 is configured correctly and is running.
290 
291 To stop the BES use the bes control script with the stop option:
292 
293 ```
294  [jimg@zoe sbin]$ besctl stop
295 ```

309 # Various features the BES supports
310 Here we highlight some important features of the BES software.
311 This list is far from comprehensive and it is not intended to be
312 a 'new features' list. Most of these features are supported by the
313 BES because they are important to the Hyrax data server. However,
314 all of these are accessible when the BES is used by itself or in
315 conjunction with other software.
316 
317 ## DAP2 versus DAP4
318 The code on the master branch now supports both DAP2 and DAP4
319 responses, and the data format handlers do as well. Because of that,
320 it must be linked with libdap 3.14, not libdap older versions of
321 libdap (libdap 3.14 contains support for both DAP2 and DAP4, while the
322 older 'libdap' supports only DAP2). If you need to get the DAP2-only
323 version of the software, use the branch named 'dap2'. Please commit
324 only fixes there.
325 
326 Each of the handlers, which can be built as submodules within this
327 code, also has DAP2 and DAP4 support on their master branch and a
328 'dap2' branch for the DAP2-only code.
329 
330 ## Support for data stored on Amazon's S3 Web Object Store
331 Hyrax 1.16 has prototype support for subset-in-place of HDF5 and NetCDF4
332 data files that are stored on AWS S3. See the preliminary documentation in
333 https://github.com/OPENDAP/bes/blob/master/modules/dmrpp_module/data/README.md.
334 
335 The new support includes software that can configure data in S3 and on
336 disk so that it can be served (and subset) in-place from S3 without reformatting
337 the original data files. Support for other web object stores besides S3
338 has also been demonstrated.
339 
340 ## Support for dataset crawler/indexer systems
341 We have added support for Datasets served by Hyrax now provide
342 information Google and other search engines need to make these data
343 findable. All dataset landing pages and catalog navigation
344 (contents.html) pages now contain embedded json-ld which crawlers such
345 as Google Dataset Search, NSF's GeoCODES, and other data sensitive web
346 crawlers use for indexing datasets. In order to facilitate this,
347 certain steps can be taken by the server administrator to bring the
348 Hyrax service to Google (and other) crawlers attention. LINK TO
349 JSON-LD README.MD. Our work on JSON-LD support was funded by NSF Grant
350 number 1740704.
351

359 ## Site-specific configuration
360 The BES uses a number of configuration files, and until now, a site has to
361 customize these for their server. Each server installation would overwrite
362 those files. No more. Now you can put all your configuration values in
363 
364 > /etc/bes/site.conf
365 
366 And be configent that they will never be overwritten by a new install. The
367 `site.conf` file is always read last, so parameters set there override values
368 set elsewhere.

370 ## The Metadata Store (MDS)
371 A new cache has been added to the BES for Metadata Responses (aka, the MDS
372 or Metadata Store). This cache is unlike the other BES caches in that it is
373 intended to be operated as either a 'cache' or a 'store.' In the latter case,
374 items added will never be removed - it is an open-ended place where metadata
375 response objects will be kept indefinitely. The MDS contents (as a cache or
376 a store) will survive Hyrax restarts.
377 
378 The MDS was initially built to speed responses when data are stored on high-
379 latency devices like Amazon's S3 storage system. We have special features in
380 Hyrax to handle data kept on those kinds of data stores and the MDS can provide
381 clients with fast access to the metadata for those data very quickly. After
382 our initial tests of the MDS, we decided to make it a general feature of the
383 server, available to data served from data stored on spinning disk in addition
384 to S3.
385 
386 Note: The MDS is not used for requests that using server-side function processing.
387 
388 The MDS configuration can be found in the dap.conf configuration file. Here
389 are the default settings:
390 
391 * Setting the 'path' to null disables uses of the MDS.
392 
393 > DAP.GlobalMetadataStore.path = @datadir@/mds
394 > DAP.GlobalMetadataStore.prefix = mds
395 
396 * Setting 'size' to zero makes the MDS hold objects forever; setting a positive
397 non-zero size makes the MDS behave like a cache, purging responses when the
398 size is exceeded. The number is the size of the cache volume in megabytes.
399 
400 > DAP.GlobalMetadataStore.size = 200
401 
402 * The MDS writes a ledger of additions and removals. By default the ledger is
403 kept in 'mds_ledger.txt' in the directory used to start the BES.
404 
405 > DAP.GlobalMetadataStore.ledger = @datadir@/mds_ledger.txt

407 ## COVJSON Responses

413 ## Improved catalog support
414 It has long been possible to define 'catalogs' for data that reside on other
415 kinds of data stores (e.g., relational data base systems). The datasets defined
416 by these catalogs appear(ed) in the directory listings as if they are
417 directories of data files, just like datasets that actually are files on a
418 spinning disk.
419 
420 We have generalized this system so that it is much easier to use. As an example
421 of new Catalog sub-system's ease of use, we have implemented a new module that
422 reads information about datasets from NASA's Common Metadata Repository (CMR)
423 and uses that to display a Virtual Directory for NASA data, where the hierarchical
424 relationships between data files is derived entirely from data read a CMR-support
425 web API.
426 
427 This software is currently available in source form only - contact us if you
428 would like to extend the BES Catalog system for your own data collections.
429 To build Hyrax with this feature enabled, build either in developer mode
430 (./configure ... --enable-developer) or using the spepcial configuration option
431 --enable-cmr.
